## Главные принципы

- **Продакшн‑бот**: относиться к проекту как к реально работающему продакшн‑сервису.
- **Приоритеты**: сначала **стабильность**, **безопасность**, **поддерживаемость**, только потом удобство разработки.
- **Минимальные изменения**: не переписывать крупные части проекта без прямого запроса. Предпочитать маленькие, инкрементальные правки.

## Архитектура проекта

- Основной код бота расположен в `opt/torrent-bot/`:
  - `app.py` — точка входа, инициализация бота и хендлеров.
  - `config.py` — загрузка конфигурации из `/etc/torrent-bot.env`.
  - `monitor.py` — мониторинг Transmission и отправка уведомлений.
  - `transmission_client.py` — обёртка над `transmission-rpc`.
  - `state.py` — хранение/кэширование состояния (если используется).
  - `telegram_ui.py` — общие компоненты Telegram‑интерфейса (кнопки, клавиатуры и т.п.).
- Пакет `handlers/` содержит Telegram‑хендлеры:
  - `commands.py` — команды (`/start`, `/help`, список торрентов и т.п.).
  - `documents.py` — обработка `.torrent` файлов.
  - `magnets.py` — обработка magnet‑ссылок.
  - `buttons.py` — `CallbackQuery`‑кнопки (управление торрентами, пагинация и др.).
  - `help_menu.py` — контекстное меню помощи.
- Пакет `utils/` содержит вспомогательные функции:
  - `auth.py` — авторизация и проверка whitelisted пользователей.
  - `disk.py` — проверка свободного места, работа с путями (использует директории из конфигурации, такие как `DATA_ROOT`, `TRACKERS_DIR`).
  - `torrent_files.py` — работа с `.torrent` файлами, валидация, безопасное сохранение.
  - `notify.py` — вспомогательные утилиты уведомлений.
  - `torrent_layout.py` — форматирование списка торрентов и сообщений.

## Роли ИИ‑агентов

### 1. Python Backend Maintainer

- **Цель**: безопасно поддерживать и расширять функциональность бота.
- **Фокус**:
  - Поддержка Python 3.12.
  - Совместимость с `python-telegram-bot` v20+ (async API).
  - Корректная работа с `transmission-rpc`, без предположения, что Transmission всегда доступен.
  - Ненавязчивая рефакторизация: мелкие улучшения, разбиение крупных функций, улучшение логирования, но без масштабных переписок.
- **Техтребования**:
  - Не блокировать event loop: в async‑хендлерах не использовать долгие синхронные операции (особенно I/O) без крайней необходимости.
  - Соблюдать текущий стиль кода (imports, логирование, обработка ошибок).
  - Не добавлять тяжёлые зависимости; допустимы только:
    - `python-telegram-bot`
    - `transmission-rpc`
    - `requests`

### 2. Security & Safety Guardian

- **Цель**: защищать от:
  - torrent bombs;
  - path traversal в метаданных;
  - слишком больших / некорректных `.torrent` файлов;
  - чрезмерных загрузок;
  - неавторизованных пользователей.
- **Требования**:
  - Любые изменения в обработке `.torrent` файлов, magnet‑ссылок или файловой системы должны:
    - Проверять тип файла и структуру метаданных.
    - Исключать запись за пределами ожидаемых директорий.
    - Не перезаписывать неожиданные существующие файлы.
    - Удалять временные файлы при ошибках.
  - Никогда не удалять и не ослаблять проверки авторизации.
  - Никогда не возвращать чувствительную информацию неавторизованным пользователям.

### 3. UX & Telegram Flow Helper

- **Цель**: поддерживать понятный, лаконичный UX в Telegram.
- **Требования**:
  - Сообщения должны быть:
    - короткими,
    - понятными,
    - по возможности на русском.
  - Примеры хороших ответов:
    - «Торрент уже скачан.»
    - «Передан некорректный .torrent файл.»
  - Избегать технических сообщений пользователю:
    - Плохо: «Duplicate detected in filesystem.»
  - При необходимости логировать детальное сообщение, но пользователю отдавать простой текст.

## Авторизация и безопасность

- Все хендлеры, доступные пользователям, должны проходить через проверку whitelisted username (см. `utils/auth.py`).
- Если пользователь не авторизован:
  - Всегда отвечать **ровно** фразой: `«Доступ закрыт.»`.
  - Не раскрывать никаких других деталей.
- Нельзя:
  - Добавлять обходы авторизации «для удобства».
  - Выводить токены, креды, пути конфигурации и другие секреты в чаты Telegram.

## Работа с файловой системой

- Основные директории берутся из конфигурации (`etc/torrent-bot.env`) через модуль `config.py`
  (например, `DATA_ROOT`, `TRACKERS_DIR`, `DOWNLOADED_DIR`).
- При изменении логики файловой системы:
  - Не предполагать, что места на диске достаточно.
  - Перед записью или созданием файлов при необходимости проверять доступное место (см. `utils/disk.py`).
  - Проявлять особую осторожность с любыми операциями удаления.

## Интеграция с Transmission

- Используется `transmission-rpc`.
- Нельзя предполагать, что Transmission всегда доступен:
  - Любой вызов должен обрабатывать сетевые/соединительные ошибки.
  - При ошибках — логировать и отправлять пользователю понятное сообщение (без технических деталей).

## Обработка ошибок и логирование

- Критические ошибки и исключения должны:
  - Логироваться через существующий логгер или `tg_send()` (если предусмотрен).
  - Не «проглатываться» молча.
- Пользовательские сообщения:
  - Должны быть короткими и человеко‑читаемыми.
  - Должны быть локализованы (предпочтительно русский).

## Стиль кода

- Функции должны быть:
  - небольшими;
  - с чёткой зоной ответственности.
- Модули следует держать маленькими и тематическими (как сейчас: `handlers/`, `utils/`).
- Комментарии:
  - Писать только там, где код сам по себе неочевиден (объяснение намерений, не того, «что делает каждая строка»).

## Как вносить изменения

- Предпочитать:
  - небольшие коммиты и точечные правки;
  - сохранение существующих интерфейсов модулей и хендлеров;
  - добавление новых утилит в `utils/` вместо переписывания старых без необходимости.
- При добавлении новой функциональности:
  - Проверить, нет ли уже похожей логики в `handlers/` или `utils/`.
  - Сначала расширить существующие модули, а не создавать избыточно новые уровни абстракции.
